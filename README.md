# Robot
Трейдинговый робот на Python с микросервисной архитектурой для Bybit. Код содержит много заделов под совместимость с другими биржами и стратегиями. 

Этот репозиторий создан для демонстрации навыков Backend-разработки работодателям. 

## Что используем?
**Используемые библиотеки и технологии:**
- NATS
- Pybit-asyncio
- PostgreSQL
- SQLAlchemy
- Стек Grafana
- Docker-compose
- Python 3
- Poetry
- Nginx
- FastAPI

Для торговли используются *ренко-свечи*. От классических свечек с фиксированной длительностью (1 мин, 5 мин, 1 день и т.д.) они отличаются фиксированным размером (+-1%, +-2% и т.п.). Они позволяют значительно сократить шум и "сжать" в одну свечу длинный боковик. Новая зелёная свеча строится от верхней точки предыдущей свечи, а новая красная свеча - от нижней точки предыдущей. То есть при развороте формируется гэп размером с одну свечу. Это нужно для избежания формирования "забора" вида [+1, -1, +1, -1 ...].

Ещё на этих свечах ML-модели учатся гораздо лучше, так как рыночные данные превращаются в дискретизированную бинарную последовательность. Суммарный объём исторических данных получается сильно меньше, а значит занимает меньше RAM при обучении моделей. Помимо ML, на ренко-свечах можно пытаться строить какие-нибудь вероятностные стратегии.

Ренко-свечи собираются из сделок. Проблемным местом этих свечек является возможность формирования сразу нескольких свечек одной сделкой. То есть их длительность будет равна нулю. Это затрудняет анализ и требует вводить дополнительные фильтры, поскольку будет физически невозможно среагировать на эти свечи. Но это скорее проблема резких скачков цены в отсутствие ликвидности, а не ренко-свечей.

По ренко-свечам строятся индикаторы теханализа и отдаются в классификаторы CatBoost, откуда получаем прогнозы точек входа и выхода.

Пока не дописал тг-бота для просмотра статистики, но он будет вместе с Redis для кэширования запросов к БД.

## Установка
Сперва заполнить все файлы .env (По одному на одном уровне с docker-compose.yml, в большинстве сервисов внутри stateless). Значение каждой настройки описано в файлах.
Затем произвести следующие команды:
```bash
cd infrastructure && docker-compose up -d
cd ../stateful && sudo docker-compose up -d
cd ../stateless && sudo docker-compose up -d --build
```
Последнюю следует выполнять спустя минут 5, пока статус stateful_postgres не станет healthy. Иногда database-worker запускается только со 2 раза, так как не может найти nats в
локальной сети. Пока не разобрался с причиной.

## Trade Stream Service
Модуль для обработки стрима сделок и ордеров Bybit. Подписывается на соответствующие стримы, валидирует сообщения и передаёт данные в Робота. Использует мою модификацию библиотеки pybit-asyncio, вносящую нужный мне функционал и устраняющую баги вендора.

## Робот
Модуль, осуществляющий всю остальную логику взаимодействия с биржей и торговли. Работает следующим образом:

1. NATSConsumer подписывается на стримы сделок и ордеров, в качестве callback'ов назначаются методы объекта Robot. 
2. Первое сообщение со сделкой запускает Initializer, добывающий данные о тикере и историю свечек. После завершения инициализации тикера сделки обрабатываются в обычном режиме: сделка передаётся в TickerProcessor, он генерирует ренко-свечи, обновляет трейлинг-стоп, считает индикаторы и предикты на покупку и продажу. 
3. Все полученные данные передаются в DecisionMaker, применяющий фильтры и выдающий окончательное решение (Buy, Sell, Update, Ignore, Delete). Собранные свечи отправляются в NATS JetStream на сохранение в базу данных.
4. Полученные решения передаются в PositionManager. Этот модуль проверяет число и статусы открытых позиций и генерирует ордер, соответствующий принятому решению.
5. Созданный ордер передаётся в OrderManager, который отправляет его на биржу.
6. Сообщения об исполненных ордерах приходят в callback для стрима ордеров. Он обращается в PositionManager для изменения локального состояния позиции.

## DatabaseWorker
Этот микросервис принимает ренко-свечи и ордера из NATS JetStream. Свечи сохраняет в БД сразу, а ордера группирует по сделкам робота и сохраняет их вместе в БД. Сделкой признаётся совокупность ордеров, которые открыли, закрыли и изменяли позицию. Использует SQLAlchemy.

## Адаптер метрик
В микросервисы к любой корутине можно добавить декоратор с вызовом методов объектов из файла packages/MonitorTools. Эти методы собирают метрики и отправляют их через NATS Core в адаптер метрик. Адаптер преобразует их в структуры данных Prometheus и отдаёт в Prometheus по его запросу через FastAPI.

## Инфраструктура
Сюда входят микросервисы Prometheus, cAdvisor, Grafana, Promtail и Loki. Prometheus забирает метрики у адаптера метрик и отдаёт их в Grafana. Promtail собирает логи из всех контейнеров, они централизованно собираются в Loki, а он уже отдаёт их в Grafana. Grafana используется для просмотра логов и визуализации собранных метрик. cAdvisor собирает "железные" метрики контейнеров - потребление CPU и RAM и отдаёт эти данные в Prometheus, а он - в Grafana.

   То есть: на отдельном сервере будет работать робот под одну биржу. На нём сидят Trade Stream Service, NATS Core, Робот, Адаптер метрик, агент cAdvisor и агент Promtail. Один центральный сервер с телеграм-ботом и всеми агрегаторами метрик и логов. Ещё как минимум один сервер должен быть выделен под работу с базой данных. Для управления пакетами, тестами и разработкой хорошо бы иметь GitLab, но он ест более 14 GB RAM, я решил подождать с его внедрением.

2. Помимо вышеописанного, текущий код Робота имеет множество потенциальных проблем с совместимостью с другими биржами, поскольку я не изучал их устройство. Bybit был в приоритете. Ещё ООП логика реализована далеко не везде (а могла быть реализована везде) и без переписывания кода с нуля этого не исправить. 

3. Код рассчитан исключительно на обработку стрима сделок и работу с ренко-свечами. Есть проблемы с совместимостью с другими стратегиями.
4. Передача ордеров с биржи осуществляется по NATS Core, который работает по принципу at-most-once, так что если упадёт Робот, ордер не будет доставлен. В целом это не сложно исправить, позже займусь.
5. В планах добавить Роботу репликацию на случай прерывания работы. Для этого надо подтянуть актуальное состояние позиций и тикеров из БД. Позже займусь.
